# アーキテクチャとトレードオフ

## 依存関係の方向性

ドメインはアプリケーション、インフラストラクチャ、CLIを参照しません。アプリケーションはドメインだけを参照します。インフラストラクチャがポートを実装し、CLIがすべてを組み立てます。

## トランザクションの境界

単一のDB内で複数の書き込みをアトミックに行う必要がある場合、トランザクション機能をアプリケーションのポートとして渡し、コントローラーが操作の境界を決めます。

PaymentとDBのような分散効果は、通常のDBトランザクションだけではアトミックになりません。次の設計を比較します。

- Paymentを先に行い、保存失敗時にリファンドや補償を行う
- 保存を先に行い、アウトボックスからpaymentを非同期実行する
- Sagaやプロセスマネージャーで進捗をドメインの状態として持つ

このラボのサンプルは小ささを優先し、Payment成功後にインメモリで保存します。本番設計ではアウトボックス、冪等性キー、リトライ、補償が必要です。

## Credentialの境界

Paymentアダプターは処理に`CardToken`を必要としますが、ロギングやドメインの結果に対して生の値を流しません。サンプルのコンソールアダプターもトークンを`[redacted]`として表示します。

本番のアダプターでは、クレデンシャルを外部プロバイダーへ渡す箇所を境界内に限定し、構造化ログや例外メッセージ、メトリクスラベルに含めないでください。診断にはトランザクションIDやプロバイダー側の安全な相関IDを使用します。

## フォルダ戦略

上位はバウンデッドコンテキストや機能で分け、内部はドメイン、アプリケーション、インフラストラクチャの依存ルールを維持します。共通エンティティを無理に共有しません。同じ「Customer」でも、請求とサポートでは必要な意味が異なります。
